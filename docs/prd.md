# Product Requirements Document (PRD)
## 대학교 데이터 시각화 대시보드

---

## 1. 제품 개요

### 1.1 제품명
대학교 실적 데이터 시각화 대시보드 (University Performance Data Visualization Dashboard)

### 1.2 제품 목적
대학교 내부의 핵심 성과 지표(실적, 논문 게재 수, 학생 수, 예산 데이터)를 웹 기반 대시보드를 통해 시각화하여, 의사결정권자와 관계자들이 데이터를 쉽고 빠르게 파악할 수 있도록 지원합니다.

### 1.3 핵심 가치 제안
- **직관적인 데이터 시각화**: 복잡한 엑셀 데이터를 막대그래프, 라인 차트, 파이 차트 등 다양한 형태로 시각화
- **간편한 데이터 관리**: 엑셀 파일 업로드만으로 자동으로 데이터 파싱 및 저장
- **접근성 향상**: 언제 어디서나 웹 브라우저를 통해 데이터 확인 가능
- **보안**: 허가된 사용자만 접근 가능한 회원 관리 시스템

### 1.4 주요 기능
1. 회원가입 및 로그인 (인증/인가)
2. 엑셀 파일 업로드 및 데이터 파싱
3. 다차원 데이터 시각화 (실적, 논문, 학생 수, 예산)
4. 대시보드 종합 뷰
5. 데이터 추이 분석 및 비교

## 2. Stakeholders

### 2.1 Primary Stakeholders
| 역할 | 책임 | 주요 관심사 |
|------|------|------------|
| **대학 경영진** | 전략적 의사결정 | 전체 실적 추이, 예산 효율성, 성과 지표 |
| **학과장/부서장** | 부서 성과 관리 | 소속 부서의 논문 게재 수, 학생 수 추이, 예산 사용 현황 |
| **행정 관리자** | 데이터 관리 및 업로드 | 데이터 정확성, 업로드 편의성, 시스템 안정성 |

### 2.2 Secondary Stakeholders
| 역할 | 책임 | 주요 관심사 |
|------|------|------------|
| **교수/연구자** | 연구 성과 확인 | 논문 게재 추이, 연구 실적 비교 |
| **기획·평가팀** | 대학 평가 대응 | 각종 평가 지표 데이터 추출 및 분석 |
| **IT 관리자** | 시스템 운영 및 유지보수 | 시스템 안정성, 보안, 성능 |

### 2.3 External Stakeholders
- **교육부/평가기관**: 대학 평가 자료 요청 시 활용
- **감사 기관**: 예산 집행 현황 확인

## 3. 포함 페이지

### 3.1 인증 페이지

0. **루트 페이지** (`/`)
   - 미인증 사용자: 로그인 폼 표시 (`/login`과 동일한 화면)
   - 인증된 사용자: 대시보드(`/dashboard`)로 자동 리디렉션
   - 사유: 사용자 편의성 - URL 입력 시 즉시 로그인 또는 대시보드 접근

1. **로그인 페이지** (`/login`)
   - 이메일/ID 및 비밀번호 입력
   - "비밀번호 찾기" 링크
   - 회원가입 페이지로 이동 링크
   - (참고: `/`와 `/login`은 동일한 로그인 화면 렌더링)

2. **회원가입 페이지** (`/signup`)
   - 사용자 정보 입력 (이름, 이메일, 소속, 직책 등)
   - 이메일 인증
   - 관리자 승인 대기

### 3.2 메인 대시보드
3. **대시보드 홈** (`/dashboard`)
   - 전체 KPI 요약 카드 (실적, 논문 수, 학생 수, 예산)
   - 최근 업데이트된 데이터 표시
   - 주요 추이 그래프 (연도별, 학기별)
   - 빠른 이동 메뉴

### 3.3 데이터 관리 페이지
4. **데이터 업로드 (Django Admin 활용)** (`/admin`)
   - 행정 관리자가 Django 관리자 페이지(/admin)에 직접 접속하여 파일 업로드
   - Django Admin의 기본 파일 업로드 위젯 활용
   - 업로드 이력 자동 기록
   - (사유: MVP 개발 속도 극대화를 위해 별도 UI 개발을 지양하고, Django의 강력한 기본 기능 활용)
   - (2단계 로드맵: 사용자 친화적인 드래그 앤 드롭 업로드 페이지 개발 예정)

### 3.4 시각화 페이지
5. **실적 시각화 페이지** (`/analytics/performance`)
   - 부서별/학과별 실적 비교 (막대 그래프)
   - 연도별 실적 추이 (라인 차트)
   - 필터링 옵션 (기간, 부서, 카테고리)

6. **논문 게재 시각화 페이지** (`/analytics/publications`)
   - 학과별 논문 게재 수 (막대 그래프)
   - 연도별 논문 추이 (라인 차트)
   - 저널 등급별 분포 (파이 차트)
   - 교수별 논문 게재 현황

7. **학생 수 시각화 페이지** (`/analytics/students`)
   - 학과별 재학생 수 (막대 그래프)
   - 학년별 분포 (파이 차트)
   - 입학/졸업 추이 (라인 차트)
   - 재적률 분석

8. **예산 데이터 시각화 페이지** (`/analytics/budget`)
   - 부서별 예산 배분 (파이 차트)
   - 연도별 예산 집행률 (라인 차트)
   - 항목별 예산 사용 현황 (막대 그래프)
   - 예산 대비 실적 비교

### 3.5 사용자 관리 페이지
9. **사용자 프로필** (`/profile`)
   - 개인정보 수정
   - 비밀번호 변경
   - 알림 설정

10. **관리자 페이지** (`/admin`) - 관리자 전용
    - 사용자 승인/거부
    - 권한 관리
    - 시스템 로그 조회

## 4. 사용자 여정 (User Journey)

### 4.1 타겟 유저 Segment 1: 대학 행정 관리자

**페르소나**: 김민수 (39세, 기획처 실장)
- 매월 초 각 부서로부터 받은 엑셀 데이터를 시스템에 업로드
- 데이터 정확성 검증 및 관리
- 경영진에게 리포트 제공

#### 여정 단계

**1단계: 로그인** (`/login`)
- 행정 관리자가 이메일과 비밀번호로 로그인
- 2FA 인증 (선택적)

**2단계: 대시보드 확인** (`/dashboard`)
- 로그인 후 자동으로 대시보드로 이동
- 전체 KPI 한눈에 확인
- 이상치 또는 누락된 데이터 확인

**3단계: 데이터 업로드** (`/admin`)
- Django 관리자 페이지(/admin) 접속
- 해당 데이터 모델(학과별 KPI, 논문 게재, 연구비 집행, 학생 명단)의 "추가" 버튼 클릭
- 파일 업로드 필드에서 엑셀/CSV 파일 선택
- 시스템이 자동으로 데이터 파싱 및 검증
- 오류 발생 시 Django Admin의 오류 메시지 표시
- 업로드 완료 확인
- (사유: MVP 개발 속도 극대화를 위해 Django의 강력한 기본 기능 활용)

**4단계: 데이터 검증** (`/analytics/*`)
- 각 시각화 페이지를 순회하며 데이터 정확성 확인
- 실적, 논문, 학생 수, 예산 페이지 각각 확인
- 필요 시 데이터 재업로드

**5단계: 리포트 준비** (`/dashboard`)
- 대시보드에서 경영진용 요약 정보 확인
- 필요 시 차트 이미지 내보내기 (향후 기능)

### 4.2 타겟 유저 Segment 2: 대학 경영진

**페르소나**: 박영희 (52세, 부총장)
- 분기별 전체 대학 성과 모니터링
- 의사결정을 위한 데이터 기반 인사이트 필요
- 기술에 익숙하지 않음, 직관적인 UI 선호

#### 여정 단계

**1단계: 로그인** (`/login`)
- 간단한 로그인 프로세스

**2단계: 대시보드 조회** (`/dashboard`)
- 전체 성과 지표 요약 카드 확인
- 주요 추이를 한눈에 파악
- 관심 지표 클릭하여 상세 페이지로 이동

**3단계: 실적 분석** (`/analytics/performance`)
- 부서별 실적 비교
- 전년 대비 성장률 확인
- 우수 부서 파악

**4단계: 예산 효율성 확인** (`/analytics/budget`)
- 예산 대비 실적 비교
- 예산 집행률 확인
- 비효율적인 예산 사용 파악

**5단계: 논문 성과 확인** (`/analytics/publications`)
- 학과별 논문 게재 추이 확인
- 연구 성과 우수 학과 파악

**6단계: 로그아웃**
- 데이터 확인 후 로그아웃

---

### 4.3 타겟 유저 Segment 3: 교수/연구자

**페르소나**: 이정훈 (45세, 컴퓨터공학과 교수)
- 본인 및 학과의 논문 게재 추이 확인
- 다른 학과와의 비교를 통한 동기부여
- 학생 수 추이를 통한 학과 발전 모니터링

#### 여정 단계

**1단계: 로그인** (`/login`)
- 교수 계정으로 로그인

**2단계: 대시보드 확인** (`/dashboard`)
- 전체 대학 성과 간략히 확인

**3단계: 논문 게재 현황 조회** (`/analytics/publications`)
- 본인 학과의 논문 게재 수 확인
- 연도별 추이 분석
- 타 학과와 비교

**4단계: 학생 수 추이 확인** (`/analytics/students`)
- 학과 재학생 수 추이 확인
- 학년별 분포 파악
- 입학률 및 졸업률 확인

**5단계: 로그아웃**

│   │

│   │   ├── 학과별 논문 수 (막대 그래프)

│   │
│   └── 예산 분석 (/analytics/budget)
│       ├── 부서별 예산 배분 (파이 차트)

### IA 설명

#### Depth 1: 주요 섹션
- **인증**: 로그인 및 회원가입 관련 페이지
- **메인 대시보드**: 전체 데이터 요약 및 빠른 접근
- **데이터 관리**: 데이터 업로드 및 히스토리 관리
- **데이터 분석 및 시각화**: 4가지 핵심 지표별 시각화 페이지
- **사용자 관리**: 프로필 및 관리자 기능

#### Depth 2: 세부 페이지
각 주요 섹션 아래 기능별로 페이지가 구성되며, 일관된 레이아웃과 네비게이션 제공

#### Depth 3: 페이지 내 컴포넌트
- 필터링 옵션
- 다양한 차트 타입
- 데이터 테이블
- 내보내기 기능

## 6. 기술 요구사항

### 6.1 기능 요구사항

#### 6.1.1 인증 및 권한 관리
- 이메일 기반 회원가입
- 안전한 비밀번호 정책 (최소 8자, 특수문자 포함)
- 관리자 승인 기반 회원 활성화
- 역할 기반 접근 제어 (RBAC)
  - 관리자: 모든 기능 접근
  - 경영진: 모든 데이터 조회
  - 일반 사용자: 제한된 데이터 조회

#### 6.1.2 데이터 업로드 및 처리
- 엑셀 파일 형식 지원 (.xlsx, .xls)
- 파일 크기 제한: 최대 50MB
- 자동 데이터 파싱 및 검증

- id

- id
- department
- professor_name

- journal_name
- journal_grade
- year
- created_at

#### Student (학생 수 데이터)
- id

- count
- created_at

#### ResearchProject (연구 과제 정보)
- id
- project_number (과제번호, 예: NRF-2023-015)
- project_name (과제명)
- principal_investigator (연구책임자)
- department (소속학과)
- funding_agency (지원기관, 예: 한국연구재단)
- total_budget (총연구비)
- created_at
- updated_at

#### ExecutionRecord (연구비 상세 집행 내역)
- id
- execution_id (집행ID, 예: T2301001)
- project (어떤 과제에 속하는지, FK → ResearchProject)
- execution_date (집행일자)
- expense_category (집행항목, 예: '연구장비 도입', '인건비', '시약 및 재료비')
- amount (집행금액)
- status (상태, 예: '집행완료', '처리중')
- description (비고/적요)
- created_at

**중요:** 상세 집행 내역(ExecutionRecord) 데이터를 동적으로 집계(Aggregation)하여 과제별 총 예산, 집행률 등의 통계를 계산하여 시각화 페이지에 표시합니다. 이를 통해 실제 CSV 데이터 구조와 일치하는 유연한 데이터 모델을 구현하며, 예산 분석 시각화 기능의 요구사항을 충족합니다.
